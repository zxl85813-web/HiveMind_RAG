# 信息源优先级指南（Source Priority Guide）

## 核心原则

> 不是所有信息都值得带进新窗口。必须按价值密度分层，把稀缺的上下文 Token 预算花在刀刃上。

---

## 二、价值密度分层规则

读取任何信息源时，必须同步执行分层：

### 🔴 核心长期信息（Core Long-term）
**特征**：在多次对话中重复出现、被明确强调、影响所有后续决策
- 架构规则与技术选型（如「用 Neo4j 存图谱」「用 Chroma 存向量」）
- 用户稳定偏好（如「不喜欢流水账总结」「重视深度连续性」）
- 项目方法论（如「五步迁移法」「TDD 微任务循环」）
- 已确定的禁区（被明确说过「不要这样做」的模式）

**处理**：全量保留，纳入迁移报告的「高权重概念」段

### 🟡 中层任务信息（Mid-layer Task）
**特征**：当前 Sprint 相关，完成后即失效
- TODO.md 中 `[ ]` 状态的未完成任务
- 当前 OpenSpec Change 的进度
- 正在调试的 Bug 与复现路径

**处理**：保留状态信息（做了什么/还剩什么），细节可精简

### 🟢 临时过程信息（Ephemeral）
**特征**：一次性的、已解决的、与当前问题不直接相关
- 已解决的错误日志
- 已完成并归档的 Task 细节
- 具体的临时测试数值

**处理**：只保留结论，丢弃过程

### ⚫ 可忽略噪声（Noise）
**特征**：无信息量的内容
- 礼貌性闲聊
- 已被明确撤销的决策
- 重复的内容（只保留最新版本）

**处理**：直接丢弃

## 三、读取顺序与 Token 预算

在 Token 有限的情况下，严格按以下顺序读取，达到阈值后停止：

```
1. P0: KI 系统摘要 + TODO.md           → 约 20% Token 预算
2. P1: 近3次对话摘要 + 活跃 REQ        → 约 30% Token 预算
3. P2: 设计文档 + ReflectionLog        → 约 30% Token 预算
4. 剩余 20% 预算留给生成迁移报告
```

> 如果 P0+P1 就已足够重建主轴，**不需要**继续读取 P2/P3。

---

## 四、阅读权重规则

读取完信息后，按以下规则为内容赋权：

| 规则 | 含义 |
|------|------|
| **高频重复** | 多次出现的概念 → 提升到「核心资产」级别 |
| **多次强调** | 被用户明确强调的偏好 → 视为稳定偏好，不可违背 |
| **反复修正** | 被修正过的内容 → 只保留**最新修正后**的版本，旧版本废弃 |
| **有定义的框架** | 已形成的公式/方法论 → 优先保留，不随意替换 |
| **深层推进方向** | 旧窗口中「这件事背后的真实目标」> 表面的聊天话题 |
