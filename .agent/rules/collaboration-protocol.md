# 🤝 HiveMind 协作协议 (Collaboration Protocol)

> **目标**: 确保开发过程的高质量、高透明度，防止 Token 溢出导致的逻辑断裂，并最大化现有功能的复用率。

## 1. 核心执行：小步慢跑 (Mini-Steps Iterator)
- **单次限制**: 除非用户明确要求“全量重构”，否则单次 `replace_file_content` 调用不得超过 **3 个相关文件**。
- **强反馈回路**: 
    - 每完成一个小阶段（如：修改完 Provider 及其关联的 Hook），需提供当前进度的微型总结。
    - 主动询问用户：“是否继续下一步？”或“这一阶段的改动是否符合预期？”
- **防溢出规约**: 
    - 在执行复杂重构时，优先保证 `tool_use`（代码写入）成功。
    - 精简非必要的冗余思考，避免在高频修改期讨论哲学问题。

## 2. 核心规约：能力注入与复用 (Capability First)
- **注册表优先**: 在实现新逻辑前，必须查阅 `REGISTRY.md` 确认是否有可复用资产。
- **监控强制**: 
    - **前端**: 所有新建/重构的页面组件必须集成 `useMonitor`。
    - **后端**: 所有核心逻辑 Service 必须从 `app.core.logging` 引入 `get_trace_logger` 代替 `loguru.logger`。
- **资产登记**: 任务完成后，若产生了新的 Service 或通用 Hook，必须引导用户或自动更新 `REGISTRY.md`。

## 3. 状态管理
- **TODO.md 联动**: 每次任务结束后，必须根据当前状态更新根目录或子模块下的 `TODO.md` 或微计划文件。
- **架构映射**: 涉及核心依赖变更时，提示需要更新 Neo4j 架构图。

---

*本协议是 Agent 与用户之间最高级别的行为准则。*
