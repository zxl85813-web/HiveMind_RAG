# 🧩 功能分解与过程日志

> 当一个功能过于复杂时，使用此流程将其分解为可管理的子任务，并记录开发过程。

## 步骤

### 1. 判断是否需要分解
功能满足以下任一条件时需要分解:
- 涉及 3 个及以上文件的变更
- 涉及前后端联调
- 包含异步流程或多步骤逻辑
- 预估开发时间超过 1 小时

### 2. 创建开发日志
在 `docs/changelog/devlog/` 创建:

```markdown
# DEV-{REQ编号}: {功能名}

| 字段 | 值 |
|------|------|
| **关联需求** | REQ-NNN |
| **开始时间** | YYYY-MM-DD HH:MM |
| **状态** | 🚧 进行中 |

## 任务分解

| # | 子任务 | 文件 | 状态 | 耗时 |
|---|--------|------|------|------|
| 1 | 定义 Schema | schemas/xxx.py | ⬜ | - |
| 2 | 实现 Service | services/xxx.py | ⬜ | - |
| 3 | 添加 API Route | api/routes/xxx.py | ⬜ | - |
| 4 | 前端 API 调用层 | services/xxxApi.ts | ⬜ | - |
| 5 | 前端组件 | components/xxx/ | ⬜ | - |
| 6 | 集成联调 | - | ⬜ | - |

## 开发日志

### Step 1: 定义 Schema
**时间**: HH:MM
**操作**: 创建了 XxxRequest, XxxResponse
**决策**: {为什么这样设计}
**问题**: {遇到的问题}

### Step 2: ...
...

## 总结
**完成时间**: YYYY-MM-DD HH:MM
**实际耗时**: X 小时
**经验教训**: ...
```

### 3. 按子任务顺序执行
每完成一个子任务:
1. 更新日志中的状态 ⬜ → ✅
2. 记录决策和遇到的问题
3. 更新 REGISTRY.md

### 4. 完成后
- 标记整体状态为 ✅
- 记录总结与经验教训
- 将日志归档
