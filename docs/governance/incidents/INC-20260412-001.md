# 🛡️ Governance Incident: [INC-20260412-001] 
# 401 鉴权失效与重定向死循环故障 (Auth Drift & Redirect Loop)

---

## 1. 事故描述 (Description)
- **触发点**: 用户登录后，部分接口（如 `/knowledge`, `/swarm/stats`）返回 401 Unauthorized。
- **症状**: 
    1. 前端由于多个并发请求同时 401，频繁调用 `tokenVault.clear()`。
    2. 产生重定向死循环（Redirect Loop），页面不断在 `/login` 刷新。
    3. 后端日志显示：部分业务接口正常（200），部分异常（401）。

## 2. 根因分析 (Root Cause Analysis)

### 2.1 后端：规约执行偏差 (Manual Dependency Bypass)
在 `backend/app/auth/permissions.py` 中，`get_current_user_local` 手动调用了 `get_current_user(None, db)`。
- **错误逻辑**: 这种手动调用跳过了 FastAPI 的 `Depends(security_scheme)` 自动注入逻辑。
- **结果**: 由于第一个参数传了 `None`，`get_current_user` 内部判定 credentials 为空，直接抛出 401。
- **影响范围**: 所有使用 `require_permission` 装饰器的端点（Knowledge, Memory 等）均受影响。

### 2.2 前端：并发处理缺乏原子性 (Race Condition)
- 当并发请求发生响应时，第一个 401 会清理 Token。第二个并发请求由于 Token 已被清理，在拦截器处理时会因缺少凭证再次触发 401 逻辑。

## 3. 治理动作 (Remediation)
- **后端修复**: 移除 `get_current_user_local`，让 `checker` 直接通过 `Depends` 执行标准鉴权流。
- **前端加固**: 引入 `window.isAuthRedirecting` 锁，确保并发场景下只执行一次重定向动作。

## 4. 进化学习：新增规约 (New Governance Rules)

基于此事故，我们将以下规则提取至 **[AGENT_CODING_RULES]**:
- **[RULE-C003] 鉴权引用闭环**: 严禁在代码中手动调用（Manual Call）带有 `Depends` 的鉴权函数并传入 `None`。必须始终通过 FastAPI 的依赖注入树进行引用。
- **[RULE-FE-001] 拦截器幂等性**: 所有的全局状态清理（如 401 清理 Token）必须具备幂等性或锁定保护，防止并发雪崩。
