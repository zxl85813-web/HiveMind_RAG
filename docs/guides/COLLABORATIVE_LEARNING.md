# 🧬 共学体系 — 蜂群的集体进化系统

> 一只蜜蜂发现了花源，不会独自享用。
> 它跳起摇摆舞，把方向和距离编码成信号，让整个蜂群受益。
> HiveMind 的共学体系，就是这支舞。

---

## 为什么要把「自省」和「协作」融为一体？

传统项目有两套割裂的机制：

- **个人学习**：各学各的，知识留在脑子里（或 Agent 的上下文里），团队无法复用
- **团队协作**：Code Review、站会、文档，但与个人的反思和成长脱节

**真正的蜂群不是这样的。** 侦察蜂发现蜜源后会通过摇摆舞共享位置；工蜂在酿蜜中发现问题会释放化学信号通知同伴。**个体的认知和群体的协作是同一个循环的两个面。**

HiveMind 共学体系将「自主查漏补缺」和「团队互相借鉴」融合为三个互锁的循环，形成一个持续进化的飞轮。

---

## 三环飞轮模型

```
                    ┌─────────────────────┐
                    │   🌍 共进循环        │
                    │  (Co-Evolution)      │
                    │                     │
                    │  知识结晶 · 技能沉淀  │
                    │  多源对标 · 体系演进  │
                    └──────┬──────┬───────┘
                           │      │
              输出汇聚 ↑    │      │    ↑ 输出汇聚
                           │      │
           ┌───────────────┘      └───────────────┐
           │                                       │
    ┌──────▼──────┐                        ┌───────▼─────┐
    │  🪞 自省循环  │   ←── 互为输入 ──→    │  🤝 互学循环  │
    │(Self-Reflect)│                        │(Mutual Learn)│
    │              │                        │              │
    │ 每日扫描      │                        │ Code Review  │
    │ 差距识别      │                        │ PR 知识流动   │
    │ 行动计划      │                        │ 角色轮转      │
    │ Reflection   │                        │ Issue 协作    │
    └──────────────┘                        └──────────────┘
```

## 第一环：自省循环（Self-Reflection Loop）

> **一个人（或一个 Agent）要先看到自己的缺口，才知道该向谁学什么。**

### 核心机制

| 机制 | 载体 | 频率 | 产出 |
|:---|:---|:---|:---|
| **每日系统扫描** | `test_daily_learning_cycle_gene0/` | 每日 | 学习报告 + 外部信号 + 风险识别 |
| **Reflection Agent** | `app/agents/` 蜂群架构 | 每次任务 | ReflectionEntry（质量/知识/设计三类） |
| **差距自检** | 学习报告中的关键风险 + 改进机会 | 每日 | 结构化差距清单 |
| **7 日行动路线** | 学习报告 D1-D7 | 每周 | 可执行的改进任务 |

### ReflectionEntry 三维分类

```
KNOWLEDGE_GAP     → 我不知道什么？       → 触发学习行动
QUALITY_ISSUE     → 我做错了什么？       → 触发修复行动
DESIGN_INSIGHT    → 我悟到了什么？       → 触发沉淀行动
```

### 自省循环的输出口

自省不是写完日记就完事。每一份自省产出都必须流向下一环：

- **KNOWLEDGE_GAP** → 写入团队共享看板（Issue/Discussion），邀请他人补充 → 进入**互学循环**
- **QUALITY_ISSUE** → 转化为可回归的测试用例 → 进入**共进循环**的知识结晶
- **DESIGN_INSIGHT** → 更新架构决策记录（ADR）或 REGISTRY.md → 进入**共进循环**的体系演进

## 第二环：互学循环（Mutual Learning Loop）

> **一只蜜蜂再优秀，它的认知半径也有限。群体智慧的关键在于低成本共享。**

### 核心机制

| 机制 | 载体 | 规则 | 学习效果 |
|:---|:---|:---|:---|
| **四视角 Code Review** | Pull Request | 逻辑 · 架构 · 测试 · 文档 | 每次 Review 都是一次跨视角学习 |
| **Issue 驱动协作** | GitHub Issues + Project Board | 拆解 → 分配 → 实现 → 审查 → 关闭 | 在做中学，互相看到思路 |
| **角色轮转** | 多角色工作流 | PO → Architect → Dev → Reviewer 定期换位 | 打破单视角盲区 |
| **PR 知识注释** | PR 描述 + Review Comment | 每个非显而易见的决策都附「为什么」 | 代码变成可学习的决策记录 |
| **差距互补配对** | 自省循环输出 | A 的 GAP 恰好是 B 的 INSIGHT → 配对学习 | 精准互补 > 随机交流 |

### 四视角 Code Review — 不只是找 Bug，而是互相学习

```
 视角一：逻辑审查          视角二：架构审查
┌──────────────┐        ┌──────────────┐
│ 实现是否正确？  │        │ 模块边界对吗？  │
│ 边界条件处理？  │        │ 有循环依赖吗？  │
│ 并发安全性？   │        │ 扩展性如何？    │
└──────────────┘        └──────────────┘

 视角三：测试审查          视角四：文档审查
┌──────────────┐        ┌──────────────┐
│ 覆盖率达标吗？  │        │ Docstring 更新？│
│ 正向 + 异常？  │        │ REGISTRY 同步？ │
│ Mock 合理吗？  │        │ TODO 更新？     │
└──────────────┘        └──────────────┘
```

**关键规则**：审查者不只说"这里有 Bug"，还要说"我会这样写，因为……"。这样 Review 本身就成了知识传递的管道。

### 差距互补配对（Gap-Insight Matching）

这是融合自省循环和互学循环的关键接口：

```
成员 A 自省报告:
  KNOWLEDGE_GAP: "不了解 Cross-Encoder 重排器的具体调优"

成员 B 自省报告:
  DESIGN_INSIGHT: "发现微调 Cross-Encoder 在专业术语上效果显著"

→ 系统自动建议：A 向 B 发起一次 15 分钟的 Pair Learning
→ 双方产出一份共享笔记 → 进入共进循环
```

## 第三环：共进循环（Co-Evolution Loop）

> **蜜蜂不只是采蜜，它们也在无意中帮花传粉。个体的学习会推动整个生态的进化。**

### 核心机制

| 机制 | 载体 | 触发条件 | 产出 |
|:---|:---|:---|:---|
| **知识结晶** | Skill 文件 / REGISTRY.md | Agent 或人类发现通用模式 | 可复用的 Skill 或组件 |
| **学习路径更新** | LEARNING_PATH.md (L0-L4) | 新模块上线 / 架构变更 | 更新后的边做边学地图 |
| **多源对标** | A/B 实验 + 厂商对比 | 每日外部信号追踪 | 技术选型更新 + 降级方案 |
| **正向飞轮** | AI 引用追踪 + 打赏机制 | 代码被 Agent 引用 | 贡献者获认可 → 更积极贡献 |
| **体系自审** | 季度回顾 | 每季度 | 治理规则自身的迭代 |

### 知识结晶流程

```
某人/Agent 在自省或 Review 中发现一个通用模式
     │
     ▼
提交 Skill 草案（SKILL.md + 脚本）到 skills/ 目录
     │
     ▼
团队/Swarm 通过 Review 验证其通用性
     │
     ▼
通过 → 注册到 REGISTRY.md →  后续任务自动复用
不通过 → 反馈改进点 → 回到自省循环迭代
```

### 正向飞轮（Positive Feedback Loop）

```
用户点赞 / Agent 引用优质代码片段
        │
        ▼
自动沉淀为 EvaluationItem（测试用例 + 参考答案）
        │
        ▼
用于 Few-shot 改进 Prompt + 检索效果回归测试
        │
        ▼
检索质量提升 + 更多点赞 → 飞轮加速
        │
        ▼
贡献者获得积分和影响力 → 更积极贡献 → 飞轮再加速
```

## 日常运转节奏

### 每日（~15-30 分钟）

| 时段 | 动作 | 对应循环 |
|:---|:---|:---|
| 开始工作前 | 浏览系统学习报告 + 外部信号 | 自省 |
| 开始工作前 | 检查 Issue 看板，领取或更新任务 | 互学 |
| 工作中 | Reflection Agent 自动质量把关 | 自省 |
| 工作中 | PR 提交时写清决策原因 | 互学 |
| 结束时 | 更新自省条目（GAP/ISSUE/INSIGHT） | 自省 |
| 结束时 | Review 至少一个同伴的 PR 或 Issue | 互学 |

### 每周（~1 小时）

| 动作 | 对应循环 |
|:---|:---|
| 回顾 7 日行动路线完成度 | 自省 |
| 差距互补配对 Pair Learning | 互学 |
| 检查是否有可结晶的通用模式 | 共进 |
| 更新 REGISTRY.md（如有新模块/Skill） | 共进 |

### 每月/季度（~2 小时）

| 动作 | 对应循环 |
|:---|:---|
| 多厂商技术对标总结 | 共进 |
| 学习路径 L0-L4 内容更新 | 共进 |
| 治理体系自身评审（规则还合理吗？） | 共进 |
| 角色轮转规划 | 互学 |

## 与现有治理体系的关系

共学体系不是一个独立的第五模块，它是**贯穿所有治理层的学习维度**：

```
┌─────────────────────────────────────────────────────┐
│                   📚 共学体系                        │
│          (贯穿所有治理层的学习维度)                     │
├──────────┬──────────┬──────────┬────────────────────┤
│ 🧭 Agent │ 🍯 数据   │ 🏭 开发  │ 🛡️ 架构           │
│  治理     │  治理     │  治理    │  治理              │
├──────────┼──────────┼──────────┼────────────────────┤
│Reflection│知识酿造   │Code      │智能路由             │
│Agent     │质量评估   │Review    │A/B 对标             │
│自动质检   │实体抽取   │Git 门禁  │熔断降级             │
│          │反馈↔自省   │SOP 工作流│Code Vault          │
├──────────┴──────────┴──────────┴────────────────────┤
│  自省循环 ↔ 互学循环 ↔ 共进循环 — 三环飞轮持续运转     │
└─────────────────────────────────────────────────────┘
```

| 治理层 | 共学体系如何嵌入 |
|:---|:---|
| **Agent 治理** | Reflection Agent 的 ReflectionEntry 是自省循环的核心数据源 |
| **数据治理** | 知识酿造过程中的质量反馈触发 QUALITY_ISSUE 自省 |
| **开发治理** | Code Review 四视角 + PR 知识注释是互学循环的主要管道 |
| **架构治理** | 多厂商 A/B 对标 + Code Vault 沉淀是共进循环的主要驱动力 |

## 落地检查清单

### 自省循环

- [ ] 配置每日学习报告自动生成（`test_daily_learning_cycle_gene0/`）
- [ ] Reflection Agent 的 ReflectionEntry 输出到结构化存储
- [ ] 每份自省报告包含 GAP / ISSUE / INSIGHT 三类条目
- [ ] 7 日行动路线在下周一回顾完成度

### 互学循环

- [ ] 所有功能开发走 Issue → Branch → PR → Review 流程
- [ ] Code Review 覆盖四个视角（逻辑/架构/测试/文档）
- [ ] PR 描述中非显而易见的决策附「为什么」说明
- [ ] 每周至少一次差距互补配对（Pair Learning）

### 共进循环

- [ ] 通用模式提交 Skill 草案并注册到 REGISTRY.md
- [ ] 每月更新学习路径 LEARNING_PATH.md
- [ ] 每月至少完成一次多厂商对标实验
- [ ] 正向飞轮机制已接入（用户反馈 → 测试用例沉淀）

## 度量指标（可选）

不强制量化，但以下指标有助于观察体系是否在运转：

| 指标 | 观测方式 | 健康标准 |
|:---|:---|:---|
| 自省活跃度 | 每周生成的 ReflectionEntry 数量 | ≥ 3 条/人/周 |
| 互学覆盖率 | PR 获得 ≥2 条实质性 Review Comment 的比例 | ≥ 60% |
| 知识结晶率 | 每月新增 / 更新的 Skill 和 REGISTRY 条目数 | ≥ 1 条/月 |
| 差距闭合度 | 上周 GAP 在本周被标记为已学习/已解决的比例 | ≥ 50% |
| 飞轮转速 | 用户正反馈沉淀为测试用例的转化率 | 持续增长趋势 |
