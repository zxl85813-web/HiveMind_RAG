## 1. 项目概述

**HiveMind RAG** 是一套内网私有化部署的企业级 AI 智能体平台。

### 它能做什么？

用一句话概括：**把企业内部的文档、数据库、业务数据，变成可以用自然语言对话查询的 AI 助手。**

具体能力包括：

| 能力 | 说明 |
|------|------|
| 📚 知识库问答 | 上传 PDF、Word、网页等文档，AI 自动理解并回答相关问题（RAG 检索增强生成） |
| 🤖 多 Agent 协作 | 多个专业 AI 助手自动分工合作，复杂任务分解后并行处理 |
| 📊 数据分析报告 | 读取数据库中的结构化数据，生成分析报告，**客户隐私数据在整个过程中不离开内网** |
| 🔌 工具扩展（Skill / MCP） | 通过标准接口对接外部系统，灵活扩展 AI 的能力边界 |
| 💰 用量监控 | 实时统计 AI 调用量与费用，30 天趋势可视化，防止超支 |

### 本次交付重点

本次主要交付了两件事：

1. **平台治理能力**（API 密钥安全管理、调用频率控制、费用上限、用量仪表盘）
2. **报价智能 Agent 示例**（展示如何在保护客户隐私的前提下，让 AI 分析销售报价数据并生成报告）

### 2.2 一次对话请求的完整旅程

以用户问"帮我分析最近的销售报价"为例：

```
① 用户在浏览器输入问题，点击发送

② 后端验证身份（登录 Token 有效）

③ 检查调用频率（防止接口被刷）和费用余额

④ 总调度 Agent 理解意图，决定由"报价分析 Agent"处理

⑤ 报价分析 Agent 执行 5 步流水线：
   - 从数据库读取最近 50 条报价
   - 把客户姓名/电话/邮件替换为代号（如 [CUST_001]）
   - 按算法选出最重要的前 N 条
   - 把代号化数据发给外部 AI 模型分析
   - 把 AI 报告里的代号替换回真实姓名/电话/邮件

⑥ 最终报告（含真实客户信息）通过流式输出实时显示在浏览器
   整个过程中，客户真实信息从未离开内网服务器
```

### 2.3 关键设计原则

| 原则 | 说明 |
|------|------|
| **隐私优先** | 敏感数据（姓名/电话/邮件）在进入 AI 模型前全部替换为代号，模型返回后再还原 |
| **费用可控** | 设置每日 Token 上限和费用上限，超限自动拦截，达到 80% 时发出预警 |
| **密钥安全** | AI 模型的 API Key 用军级加密（Fernet AES-128）存储，只写入不可读取 |
| **可扩展** | 新业务场景通过 Skill 或 MCP 接口插入，不需要改动核心代码 |

## 4. 各功能模块说明

### 4.1 API 密钥安全管理

**解决什么问题？**

接入 OpenAI、Claude 等 AI 服务需要 API Key，如果明文写在配置文件里，一旦服务器被入侵就会泄露，可能产生巨额账单。

**我们怎么做？**

- 管理员通过后台接口将 API Key **写入**系统
- 系统用军级加密算法（AES-128 对称加密 + HKDF 密钥派生）加密后存入数据库
- **该接口只能写入，不能读取**——即使有人获得了数据库访问权限，也无法还原出明文 Key
- AI 模型调用时，Key 从内存缓存中直接读取（不再查数据库），速度快且安全

---

### 4.2 调用频率控制与费用上限

**解决什么问题？**

AI 调用按 Token 计费，一旦出现程序 Bug 或恶意刷取，费用可能在短时间内爆炸性增长。

**我们怎么做？**

设置了 **6 层保护机制**，请求必须依次通过所有层才能到达 AI 模型：

```
用户请求进来
  │
  ├── 第 1 层：每秒请求数（RPS）上限    → 超出立即返回"请稍后重试"
  ├── 第 2 层：每分钟请求数（RPM）上限  → 同上
  ├── 第 3 层：系统每日 Token 总用量上限 → 超出停止服务直到次日
  ├── 第 4 层：系统每日费用总上限（$）  → 超出停止服务直到次日
  ├── 第 5 层：单个账号每日 Token 上限  → 防止个别用户耗尽资源
  └── 第 6 层：单次对话 Token 上限      → 防止超长对话失控
  
达到 80% 时：写入预警日志（不停服，仅记录）
超过 100% 时：拒绝请求，返回 HTTP 429
```

## 5. 报价智能 Agent — 完整演示

### 5.1 这个功能解决什么问题？

销售团队有大量报价记录存在数据库里，想让 AI 帮忙分析"哪些客户最值得跟进"、"最近有哪些风险信号"——但又不放心把客户的姓名、电话、邮件直接发给 OpenAI / Claude 这样的境外 AI 服务。

**我们的解法**：在数据离开服务器之前，把所有客户隐私字段替换成代号（如 `[客户_001]`），让 AI 分析代号数据；AI 返回报告后，再把代号换回真实信息。**AI 模型全程只看到代号，真实客户信息从未离开内网。**

### 5.2 流程总览

```
┌─────────────────────────────────────────────────────────────────────┐
│                      报价智能 Agent 处理流程                          │
│                                                                       │
│   ① 取数    ② 脱敏           ③ 筛选         ④ AI 分析   ⑤ 回填     │
│                                                                       │
│  数据库  →  客户信息     →  按算法选  →  AI 模型  →  最终报告        │
│  报价表     替换为代号      出前 N 条     只看代号     代号还原为     │
│             [CUST_001]                              真实姓名          │
│             [PHONE_001]                                               │
│             [EMAIL_001]                                               │
└─────────────────────────────────────────────────────────────────────┘
```

### 5.3 五步详细说明

---

#### 第一步：从数据库取原始报价数据

系统从 `quotes` 表读取最新的报价记录，**每一条都包含完整的客户信息**：

```
姓名：    Jack Brown
电话：    +1-212-555-7788
邮件：    jack@lexcorp.us
公司：    LexCorp
产品：    HiveMind RAG Enterprise
金额：    $49,900
地区：    AMER（美洲）
状态：    draft（草稿中）
日期：    2026-04-26
```

本次演示共从数据库取到 **50 条**报价记录（10 位客户 × 6 款产品）。

---

#### 第二步：隐私脱敏（TokenVault）

这是整个方案的**核心安全机制**。系统为每一个唯一的隐私值分配一个稳定代号：

> **关键规则**：同一个值永远映射到同一个代号。
> 例如，"Jack Brown"在所有 50 条记录里都变成 `[CUST_001]`，AI 能识别出"这是同一个人"，分析时保持一致。

**脱敏对照表（部分）：**

| 原始信息（内网保留） | → | 发给 AI 的代号 |
|---|---|---|
| `Jack Brown` | → | `[CUST_001]` |
| `+1-212-555-7788` | → | `[PHONE_001]` |
| `jack@lexcorp.us` | → | `[EMAIL_001]` |
| `LexCorp` | → | `[COMPANY_001]` |
| `Ivy Tanaka` | → | `[CUST_002]` |
| `+81-3-9988-7766` | → | `[PHONE_002]` |
| `ivy@wayne.jp` | → | `[EMAIL_002]` |
| `Henry Sun` | → | `[CUST_003]` |
| … | → | … |

50 条记录共生成 **36 个唯一代号**（10 位不同客户，每人姓名/电话/邮件/公司各 1 个）。

**脱敏后的记录长这样**（这才是发给 AI 的数据）：

```json
{
  "customer_name":    "[CUST_001]",
  "customer_phone":   "[PHONE_001]",
  "customer_email":   "[EMAIL_001]",
  "customer_company": "[COMPANY_001]",
  "product_name":     "HiveMind RAG Enterprise",
  "amount_cents":     4990000,
  "region":           "AMER",
  "status":           "draft"
}
```

非隐私字段（产品名、金额、地区、状态）**原样保留**，AI 可以正常分析业务数据。

---

#### 第三步：算法筛选前 N 条（Top-N 排序）

50 条数据如果全部发给 AI，费用会很高，而且 AI 不擅长处理太多数据。系统先用算法筛选出最有价值的几条。

**默认算法（金额 × 时间衰减）：**

$$\text{得分} = \text{金额} \times e^{-\text{天数} / 30}$$

- 金额越大，得分越高
- 日期越新，得分越高（每过 30 天，得分衰减到 37%）
- **结果**：价值高且新鲜的商机排在最前面

**三种可选策略：**

| 策略名称 | 说明 | 适用场景 |
|--------|------|---------|
| `amount_weighted_recency`（默认） | 金额 × 时间衰减综合得分 | 常规销售跟进 |
| `amount_desc` | 纯金额从高到低 | 关注大单 |
| `recency` | 最新日期优先 | 关注新动态 |

| #2 | [CUST_002] | HiveMind RAG Enterprise | $49,900 | 已赢单 | 2026-04-20 |

## 10. 安全设计说明

### 10.1 客户隐私保护

这是本系统最核心的安全承诺：**客户的姓名、电话、邮件永远不会发送给任何外部 AI 服务。**

| 可能的风险 | 我们的应对方式 |
|-----------|--------------|
| AI 模型接触到真实姓名/电话/邮件 | 发送前全部替换为代号，AI 只看到 `[CUST_001]` 这样的符号 |
| AI 模型"猜出"代号对应的真实人 | System Prompt 明确指令禁止猜测；即使 AI 猜出写在报告里，也不会被系统识别为有效代号，回填时会原样保留（不会误填） |
| 代号映射表泄露 | 映射关系只存在于本次请求的服务器内存中，请求结束立即销毁，不写入数据库，不记录日志 |
| 日志文件中出现客户信息 | 系统日志只记录处理条数和统计数字，不记录任何记录内容 |

### 10.2 API 密钥保护

| 可能的风险 | 我们的应对方式 |
|-----------|--------------|
| 密钥以明文存储在数据库中 | 使用军级加密（AES-128 + HKDF 密钥派生）加密后存储，数据库中只有密文 |
| 通过接口读回密钥原文 | 写入接口（PUT only），无读取接口；返回的只是密钥末 4 位提示，不是原文 |
| 服务器被入侵后密钥泄露 | 加密主密钥（`SECRET_KEY`）存在环境变量中，与数据库分离，入侵数据库无法解密 |
| 密钥需要轮换时影响服务 | 写入新值后系统自动刷新内存缓存，无需重启服务，对用户透明 |

### 10.3 接口访问控制

| 可能的风险 | 我们的应对方式 |
|-----------|--------------|
| 未登录用户访问数据 | 所有接口需要 JWT 登录 Token，Token 缺失或过期直接拒绝 |
| 高频刷取接口造成费用暴增 | 6 层频率+费用保护（见第 4.2 节），超限自动返回 429 |
| 费用超出预算 | 每日 Token 和 USD 双重上限，达到 80% 预警，100% 停服 |

### 10.4 符合国际安全标准

本系统设计参照 OWASP Top 10 安全威胁清单：

| 安全威胁类别 | 状态 |
|-------------|------|
| 访问控制失效（A01） | ✅ JWT 认证 + 管理员权限检查 |
| 加密机制失效（A02） | ✅ AES-128 + HKDF 密钥派生 |
| 注入攻击（A03） | ✅ ORM 参数绑定，无 SQL 字符串拼接 |
| 安全配置错误（A05） | ✅ 密钥必须通过环境变量配置，无内置默认值 |
| 认证失败（A07） | ✅ JWT HS256 签名验证 + 账号状态检查 |
| 安全日志缺失（A09） | ✅ 预算预警事件写入审计日志 |
